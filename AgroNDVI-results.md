### Развилка: OSM вместо ручной разметки

### Поворот: рис вместо пшеницы
Запрос Overpass на `landuse=farmland` в bbox северной части tile 37TDK (Темрюкский район) вернул 152 OSM-объекта. После фильтрации по площади (>=20 га) и отбора топ-20 по убыванию выяснилось: **14 из 20 полей имеют тег `crop=rice`**, остальные 6 без тега, но в том же кадастре -- скорее тоже рис. Это специфика Темрюкского района: исторически рисовый, орошение от Кубани и лиманов, чеки по 700-1500 га. На пшеничные районы (Калининский, Славянский) попадём, когда расширим bbox восточнее.

Адаптируемся: NDVI работает на любой культуре, методология одна и та же. Рисовый профиль NDVI даже интереснее для портфолио (редкая ниша, выраженная сезонная динамика затопления чеков).

### Интерпретация
- На 3 июня рисовые чеки **частично или полностью затоплены** (рисосеяние идёт май-июнь). Вода даёт NDVI около 0, отсюда низкие медианные значения по полям.
- F005 (0.47), F008 (0.52), F014 (0.56) -- поля с заметной растительностью, возможно, ранний рис на стадии всходов или другие культуры (тегов crop у них нет).
- F009, F016, F004 -- значения 0.17-0.20, это либо вода в чеках, либо сильно затоплены.
- Различие 0.17 -> 0.56 между полями на одной дате доказывает: zonal stats разрешает индивидуальную динамику, а не средние по району. Это и есть основа для time-series по каждому полю отдельно.

### Что доказано

- folium-карта даёт интерактивный обзор: спутниковая подложка Esri + векторные поля + popup со статистикой по клику.
- В выбранном районе доминирует рисовое земледелие, проект адаптируется к этой культуре без потери методологической ценности.

### День 4: time-series NDVI по 20 полям за полный сезон

### Развилка: pystac-client падал на пагинации
При первом запуске `find_sentinel.py --series` с интервалом 12 месяцев `pystac-client` падал с `IncompleteRead` и `ReadTimeoutError` из-за нестабильного канала РФ -> us-west-2. ```
yield = 55 ц/га + 30*(NDVI_peak - median_NDVI_peak) + N(0, σ=2)

В коде и логе явно зафиксировано: **это не реальные данные, а синтетика для демонстрации pipeline**. Разброс по 20 полям: ~47-63 ц/га, согласуется с открытыми данными по Кубани (2024: ~50-55 ц/га на районном уровне).

- **F017** -- rank 3/5/2. Самый ранний пик (doy 134 -- 13 мая). Сдвинут во времени относительно остальных.

Детектор находит и **положительные**, и **отрицательные** выбросы -- это важно для агро-страхования.

### Согласованность методов (Spearman ρ)
- Z vs L2: 0.96
- Z vs IsoForest: 0.70
- L2 vs IsoForest: 0.79

Три независимых метода дают согласованный сигнал -- это и есть устойчивая аномалия, а не артефакт одного алгоритма.

### Что доказано
- Anomaly-pipeline работает на time-series NDVI без задействования каких-либо «продвинутых» методов: z-score, IsolationForest, L2 -- весь minimum sufficient toolkit.
- Heatmap z-score визуально читается как агрономический документ: видно не только «какие поля», но и «когда» произошло отклонение.
- Anomaly score готов к интеграции с моделью прогноза (день 8): для полей в top-K к прогнозу будет добавлен флаг «out-of-distribution».

### Главное ограничение (зафиксировано в отчёте)
В индустриальной задаче коридор строится по **multi-year** данным одного поля. У нас данные за один год, коридор по соседним полям -- слабый proxy. На multi-year датасете (5 лет 2020-2024) то же `pointwise_z` пересчитается тривиально, нужны только дополнительные STAC-запросы. Это «что дальше» для портфолио.
